
# XJTU Battery Dataset - 数据整理说明 (README)

## 1. 数据集概况

* **Dataset ID**: `dataset_XJTU`
* **原始数据规模**: 包含 Batch 1-6，共计 48 个真实的物理电池文件（排除校准补偿文件）。
* **电池规格**: 力神 (Lishen) 18650 圆柱形电池，额定容量 2.0Ah，额定电压 3.6V。
* **材料体系**: NCM (正极: LiNi0.5Co0.2Mn0.3O2, 负极: Graphite)
* **包含文件**:
1. `dataset_XJTU_metadata.csv` (元数据表)
2. `dataset_XJTU_cycle_summary.csv` (周期汇总表)
3. `dataset_XJTU_timeseries.csv` (时间序列表)


## 2. 字段映射与物理量标准化

根据 `BatteryTwin Schema v0.2` 规范，对原始数据进行了以下标准化处理：

* **时间单位对齐**: 原始测试仪输出的相对时间为分钟 (`relative_time_min`)，已在时序表中统一乘以 60，转换为标准秒 (`time_s`)。
* **电流符号对齐 (放电取负)**: 严格遵循规范。原始数据中放电电流记录为正值，已在脚本中作乘 `-1` 处理，确保 `current_A` 字段下所有放电电流均为负数。
* **唯一化标识 (Cell ID)**: 原始文件在不同 Batch 中存在同名情况（如 `2C_battery-1.mat`）。为确保全局唯一性，`cell_id` 已采用 `{Batch_Name}_{File_Name}` 格式拼接（例：`Batch-1_2C_battery-1`）。

## 3. 核心工程挑战与清洗逻辑记录

*注：由于原始 XJTU `.mat` 文件的特殊结构，本数据集的 ETL (抽取、转换、加载) 过程极为复杂，已通过定制化多模块 Python 脚本解决以下工程痛点：*

1. **多层嵌套数据结构的解构**
原始数据并非扁平表格，而是 `(1, N)` 维度的 Matlab 嵌套结构体。初次处理时仅用了raw_data = mat_data['data'][0, 0]，发现可能只读取了第一个循环数据，故写了写了一个检测脚本，判断代码的维度，实际检测某一个mat文件维度为390，故脚本需要通过多重循环深入底层字典，逐一提取每一个电池生命周期内（390个cycle）的变长数组，并将其重构为长表（Long DataFrame）格式。
2. **超大时序文件的处理**
由于全寿命周期的秒级时序数据量极其庞大（数千万行），传统的全量加载方式会导致处理时间过长，初次处理时考虑了对多进程并发处理，但并未max_workers的值没有限制，结果在终端显示直接跳过数据。为此，在提取 `timeseries` 时，特别采用了**“ProcessPoolExecutor 多进程并发 + 边算边存 (增量追加模式)”**的底层架构，保守起见，max_workers的值为2或4，将巨型数组的峰值内存控制在安全范围内，可根据电脑性能调整这个值。
3. **隐性元数据的溯源与注入**
注意：原始 `.mat` 文件仅包含单纯的物理测试信号，完全缺失电池物理参数（品牌、化学体系、额定电压、测试协议等）。在处理过程中，通过溯源该数据集的原始发表文献，手动将正确的 NCM 三元材料属性及详细充放电协议（如 `CC-CV 4A to 4.2V`）注入并对齐到 `metadata.csv` 中。
4. **脚本写作逻辑**
最初处理数据的脚本统一放在etl_xjtu.py中，分三个文件，分别处理metadata（元数据），cyclesummary（循环表），timeseries（时序表），但在处理中发现，这种方式的优点是可以同时处理所有数据集cyclesummery和timeseries，一定程度上节约时间，但缺点是每次运行脚本耗时太长，每次完整运行需20-30分钟，极为漫长；且与原数据集文件并没有呈现出一一对应的关系（即一个batch对应一个脚本对应一份cyclesummary和timeseries），可视化不足。
故特将脚本与处理文件按照batch集一一对应设立文件夹，存储各自对应的循环表与时序表，现在，运行各batch的循环表时间仅需30到90秒，运行各batch中各battery的时序表总共仅需1-3分钟。（如认为手动运行所有脚本过于麻烦，可创建一个run_all.py 脚本，独立运行所有脚本）

## 4. 缺失字段说明

根据原始数据限制，部分 Schema 字段留空（填充为 `NaN` 或 `unknown`）：

* `charge_capacity_Ah` (时序表): 该批次数据主要记录放电阶段，故时序层面的充电容量为空。
* `internal_resistance_Ohm` (周期表): 原始测试未提供每个 Cycle 的内阻测量值。
